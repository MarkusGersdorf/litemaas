# LiteLLM SSL Configuration for Internal Services

This guide explains how to configure LiteLLM to work with internal Kubernetes/OpenShift services that use self-signed SSL certificates, such as KServe inference endpoints.

## Problem

When LiteLLM attempts to make HTTPS requests to internal services (e.g., `https://model-service.namespace.svc.cluster.local:8000`), it may encounter SSL certificate verification errors because these services typically use self-signed certificates generated by Kubernetes/OpenShift.

**Common error messages:**
```
SSL: CERTIFICATE_VERIFY_FAILED
SSLError: [SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed
```

## Solutions

### Option 1: Disable SSL Verification (Quick Fix - Development/Testing)

⚠️ **Security Warning**: Only use this in **development/testing** environments or when you trust the internal network completely. This disables certificate validation entirely.

#### Helm Deployment

Add to your `values.yaml`:

```yaml
litellm:
  # Disable SSL verification for internal services
  sslVerify: false
```

Then upgrade your deployment:

```bash
helm upgrade litemaas deployment/helm/litemaas/ -n litemaas -f my-values.yaml
```

#### Kustomize Deployment

Edit `deployment/kustomize/litellm-deployment.yaml` and uncomment the `SSL_VERIFY` environment variable:

```yaml
env:
  # ... other env vars ...
  - name: SSL_VERIFY
    value: 'false'
```

Then apply:

```bash
kubectl apply -k deployment/kustomize/
```

### Option 2: Mount Service CA Bundle (Production - OpenShift)

This is the **recommended approach** for production OpenShift deployments. OpenShift provides a CA bundle that includes certificates for all services using Service Serving Certificates.

#### How It Works

1. OpenShift automatically injects the service CA bundle into a ConfigMap when you add a specific annotation
2. Mount this ConfigMap into the LiteLLM container
3. Configure LiteLLM to use the CA bundle for SSL verification

#### Implementation

**For Helm deployments**, add to your `values.yaml`:

```yaml
litellm:
  # Keep SSL verification enabled
  sslVerify: true
  
  # Mount OpenShift service CA bundle
  volumes:
    - name: service-ca
      configMap:
        name: litellm-service-ca
  
  volumeMounts:
    - name: service-ca
      mountPath: /etc/ssl/certs/service-ca.crt
      subPath: service-ca.crt
      readOnly: true
  
  env:
    - name: SSL_CERT_FILE
      value: /etc/ssl/certs/service-ca.crt
```

Then create the ConfigMap with the annotation:

```bash
cat <<EOF | oc apply -f -
apiVersion: v1
kind: ConfigMap
metadata:
  name: litellm-service-ca
  namespace: litemaas
  annotations:
    service.beta.openshift.io/inject-cabundle: "true"
EOF
```

**Note**: The ConfigMap will be automatically populated by OpenShift with the service CA bundle.

**For Kustomize deployments**, the chart needs to be enhanced to support this pattern. In the meantime, manually patch the deployment:

```bash
# Create the ConfigMap
oc create configmap litellm-service-ca \
  --namespace=litemaas \
  --dry-run=client -o yaml | \
  oc annotate -f - \
  service.beta.openshift.io/inject-cabundle=true \
  --dry-run=client -o yaml | \
  oc apply -f -

# Patch the deployment
oc patch deployment litellm -n litemaas --type=json -p='[
  {
    "op": "add",
    "path": "/spec/template/spec/volumes/-",
    "value": {
      "name": "service-ca",
      "configMap": {"name": "litellm-service-ca"}
    }
  },
  {
    "op": "add",
    "path": "/spec/template/spec/containers/0/volumeMounts/-",
    "value": {
      "name": "service-ca",
      "mountPath": "/etc/ssl/certs/service-ca.crt",
      "subPath": "service-ca.crt",
      "readOnly": true
    }
  },
  {
    "op": "add",
    "path": "/spec/template/spec/containers/0/env/-",
    "value": {
      "name": "SSL_CERT_FILE",
      "value": "/etc/ssl/certs/service-ca.crt"
    }
  }
]'
```

### Option 3: Per-Model SSL Configuration (Granular Control)

If you only need to disable SSL verification for specific models, you can configure this per-model through the LiteMaaS UI or API.

When creating or editing a model in LiteMaaS, add to the model's custom parameters:

```json
{
  "verify": false
}
```

This will be passed to LiteLLM's model configuration, disabling SSL verification only for that specific model.

**Note**: This feature requires LiteLLM backend support and may not be available in all versions.

## Verification

After applying your configuration, verify SSL connectivity:

1. **Check LiteLLM logs**:
   ```bash
   kubectl logs -n litemaas deployment/litellm -f
   ```

2. **Test model connectivity** through the LiteMaaS UI:
   - Go to Admin → Models
   - Create or edit a model pointing to your internal service
   - Click "Test Configuration" to verify connectivity

3. **Make a test request**:
   ```bash
   # Get your API key from LiteMaaS UI
   API_KEY="sk-litellm-xxx"
   
   # Test the model
   curl https://litellm.apps.your-cluster.com/v1/chat/completions \
     -H "Authorization: Bearer $API_KEY" \
     -H "Content-Type: application/json" \
     -d '{
       "model": "your-model-name",
       "messages": [{"role": "user", "content": "Hello"}]
     }'
   ```

## Troubleshooting

### Still Getting SSL Errors After Disabling Verification

1. **Check environment variable is set**:
   ```bash
   kubectl exec -n litemaas deployment/litellm -- env | grep SSL_VERIFY
   ```

2. **Verify pod was restarted** after configuration change:
   ```bash
   kubectl rollout status deployment/litellm -n litemaas
   ```

3. **Check Python SSL environment** (LiteLLM uses Python's `requests`/`httpx`):
   ```bash
   kubectl exec -n litemaas deployment/litellm -- python -c "import ssl; print(ssl.OPENSSL_VERSION)"
   ```

### Service CA Bundle Not Working

1. **Verify ConfigMap was populated**:
   ```bash
   oc get configmap litellm-service-ca -n litemaas -o yaml
   ```
   
   You should see a `service-ca.crt` key with certificate data.

2. **Check annotation is present**:
   ```bash
   oc get configmap litellm-service-ca -n litemaas -o jsonpath='{.metadata.annotations}'
   ```

3. **Verify the CA bundle is mounted**:
   ```bash
   kubectl exec -n litemaas deployment/litellm -- cat /etc/ssl/certs/service-ca.crt
   ```

### Mixed Environments (Internal + External Services)

If you're using both internal services (KServe) and external services (OpenAI, Azure) in the same LiteLLM instance:

- **Do NOT disable SSL verification globally** (security risk for external services)
- **Use Option 2** (Service CA bundle) - it will validate both internal and external certs correctly
- If Option 2 doesn't work, consider using **Option 3** (per-model configuration) to disable SSL only for internal models

## Security Considerations

| Option | Security Level | Use Case |
|--------|---------------|----------|
| Option 1 (Disable SSL) | ⚠️ **Low** | Development/testing only |
| Option 2 (CA Bundle) | ✅ **High** | Production (recommended) |
| Option 3 (Per-Model) | ⚙️ **Medium** | Mixed environments with granular control |

**Best Practices**:
- ✅ Always use Option 2 (CA bundle) in production
- ✅ Limit Option 1 to trusted internal networks
- ✅ Document which models have SSL verification disabled
- ❌ Never disable SSL verification when connecting to external APIs (OpenAI, Azure, etc.)
